# Lab 0 渐进式 Hints

先独立思考，再按层展开。每一层都比上一层更接近实现；看到能够继续时就停下。

## 症状：单线程会重复或越过范围

### Hint 1

把“领取下一张牌”拆成两个问题：

1. 怎样让每次调用得到不同的编号？
2. 得到编号后，怎样判断它是否仍在 `[0, total_)`？

先取得编号，后判断范围。越过终点的线程只需返回 `false`，不必把计数器减回去。

### Hint 2

需要的是一个不可分割的 read-modify-write。分别调用 `load()` 和 `store()` 中间存在窗口，两个线程能读到同一个旧值。

### Hint 3

`fetch_add(1)` 返回加法发生前的值。把这个旧值当作候选 `morsel_id`；候选在范围内时写入引用并成功，否则失败。

## 症状：单线程通过，并发测试偶发重复

### Hint 1

检查“取编号”的全过程是否只有一个原子操作。原子变量并不会自动让围绕它的多条语句成为一个原子事务。

### Hint 2

这里不需要 CAS 重试，也不需要 mutex。多个线程越过 `total_` 没关系，因为这些编号不会被返回给调用方。

### Hint 3

`morsel_id` 是每个调用方自己的输出变量，不是共享调度状态。共享且需要原子化的只有 `next_`。

## 自检问题

- `total_ == 0` 时，第一次调用会发生什么？
- 8 个线程争抢唯一一个 morsel 时，为什么只能有一个线程看到旧值 0？
- 计数器超过 `total_` 后继续增长，为什么不影响 exactly-once 语义？
- `memory_order_relaxed` 为什么足够？提示：编号分配不负责发布其他内存。
